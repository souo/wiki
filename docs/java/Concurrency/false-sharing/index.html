java中的伪共享

CPU的缓存是以缓存行(cache line)为单位进行缓存的，当多个线程修改不同变量，而这些变量又处于同一个缓存行时就会影响彼此的性能。 例如：线程1和线程2共享一个缓存行，线程1只读取缓存行中的变量1，线程2修改缓存行中的变量2。 虽然线程1和线程2操作的是不同的变量，由于变量1和变量2同处于一个缓存行中，当变量2被修改后，缓存行失效， 线程1要重新从主存中读取，因此导致缓存失效，从而产生性能问题。

伪共享​


该图显示了在不同 CPU 上运行的两个线程，它们写入不同的变量（变量存储在同一 CPU 缓存行中）导致伪共享。

CPU三级缓存​

当CPU要读取一个数据时，首先从一级缓存中查找，如果没有找到再从二级缓存中查找，如果还是没有就从三级缓存或内存中查找。一般来说，每级缓存的命中率大概都在80%左右，也就是说全部数据量的80%都可以在一级缓存中找到，只剩下20%的总数据量才需要从二级缓存、三级缓存或内存中读取，由此可见一级缓存是整个CPU缓存架构中最为重要的部分。 下表是一些缓存未命中的消耗数据：

缓存行​

当 CPU 缓存从下层缓存或主 RAM 读取数据时（例如 L2到L1、L3到 L2及主 RAM到 L3），不会低效的一次只读取一个字节，而是读取缓存行。 一个缓存行通常由 64 个字节组成。 因此，缓存一次从较低级别的缓存或主 RAM 读取 64 个字节。

因为一个缓存行由多个字节组成，所以一个缓存行通常会存储多个变量。 如果同一个 CPU 需要访问存储在同一个缓存行中的更多变量 - 这是一个优势。 如果多个 CPU 需要访问存储在同一缓存行中的变量，则可能会发生错误共享。

缓存行无效​

当一个cpu写入缓存行时,缓存行会变脏。缓存行需要与其他也需要该缓存行的cpu进行同步。其他cpu的该缓存行变得无效（需要刷新数据）。

缓存失效后的缓存刷新可以通过缓存一致性机制发生，也可以通过从主 RAM 重新加载缓存行来进行。

在刷新之前，不允许 CPU 访问该缓存行。

缓存无效带来的性能问题​

当一个缓存行因该缓存行内的数据已被另一个 CPU 更改而无效时，则需要刷新无效的缓存行 - 无论是从 L3 缓存，还是通过缓存一致性机制。因此，如果 CPU 需要读取无效的缓存行，则必须等到缓存行刷新。这会导致性能下降。 CPU 的时间浪费在等待缓存行刷新上——这意味着 CPU 在这段时间内可以执行更少的指令。

伪共享意味着两个（或更多）CPU 正在写入存储在同一缓存行中的变量，但每个 CPU 并不真正依赖另一个 CPU 写入的值。然而，他们仍然不断地使缓存线变脏，导致它对另一个 CPU 无效，迫使另一个 CPU 刷新它 - 在另一个 CPU 也使缓存线变脏之前，导致第一个 CPU 必须刷新它等等。

伪共享的解决方案是更改数据结构，以便 CPU 使用的独立变量不再存储在同一缓存行中。

note

注意：即使 CPU 有时会使用其他 CPU 写入的变量，确保共享变量不存储在同一缓存行中仍然是十分有用。

伪共享example​

public class Counter {
    public volatile long count1 = 0;
    public volatile long count2 = 0;
}

由两个线程使用的 Counter 类。 第一个线程将增加 count1 字段，第二个线程将增加 count2 字段。

public class FalseSharingExample {

    public static void main(String[] args) {
        Counter counter1 = new Counter();
        Counter counter2 = counter1;

        long iterations = 1_000_000_000;

        Thread thread1 = new Thread(() -> {
            long startTime = System.currentTimeMillis();
            for(long i=0; i<iterations; i++) {
                counter1.count1++;
            }
            long endTime = System.currentTimeMillis();
            System.out.println("total time: " + (endTime - startTime));
        });
        Thread thread2 = new Thread(() -> {
            long startTime = System.currentTimeMillis();
            for(long i=0; i<iterations; i++) {
                counter2.count2++;
            }
            long endTime = System.currentTimeMillis();
            System.out.println("total time: " + (endTime - startTime));
        });

        thread1.start();
        thread2.start();
    }
}

在我的笔记本电脑中，这段代码执行了49秒。如果更改代码，使不同线程访问不同Counter实例，则执行时长为6秒。伪共享对性能影响是很明显的。 这种速度差异很可能是由第一个示例中的伪共享引起的，其中共享 Counter 实例中的 count1 和 count2 字段在运行时位于同一缓存行中。

在第二个示例（第二个tab）中，两个线程使用各自的 Counter 实例，它们不再将其字段存储在同一缓存行中。 因此，不会发生伪共享，并且代码运行得更快。

解决伪共享问题​

解决伪共享问题的方法是设计您的代码，以便不同线程使用的不同变量最终不会存储在同一 CPU 缓存行中。 具体如何执行取决于具体的代码，但将变量存储在不同的对象中是一种方法——如上一节中的示例所示。

使用@Contented 注解防止伪共享​

从 Java 8 和 9 开始，Java 具有 @Contended 注解，它可以用空字节填充类内的字段（在字段之后 - 当存储在 RAM 中时），以便该类的对象内的字段不会存储在同一个 CPU 缓存行。 下面是前面示例中的 Counter 类，其中一个字段中添加了 @Contended 注解。 添加此注解使执行时间下降到与两个线程使用两个不同 Counter 实例时的时间大致相同。 这是修改后的 Counter 类：

public class Counter {

    @sun.misc.Contended
    public volatile long count1 = 0;
    public volatile long count2 = 0;
}

配置 Padding Size​

默认情况下，@Contended 批注会在使用 @Contended 注解的字段之后添加 128 字节的填充。 但是，您可以通过 JVM 命令行参数告诉 Java VM 要用作填充的字节数。

-XX:ContendedPaddingWidth=64

此参数将 Java VM 配置为使用 64 字节填充 @Contended 注解，而不是 128 字节。

在 Java 中避免伪共享所需的填充字节量取决于底层硬件架构 - 意味着每个 CPU 缓存行包含多少字节。

因此，如果缓存行只有 64 字节，则没有理由填充 128 字节。 此外，如果缓存行是 256 字节，仅填充 128 字节将不足以防止伪共享。

总结​

  • CPU缓存是以缓存行为单位进行操作的。产生伪共享问题的根源在于不同的核同时操作同一个缓存行
  • 解决伪共享问题的方法是设计您的代码，以便不同线程使用的不同变量不会存储在同一 CPU 缓存行中
  • 也可以通过填充来解决伪共享问题，Java8 中引入了@Contended 注解来自动填充。
  • 并不是所有的场景都需要解决伪共享问题，因为CPU缓存是有限的，填充会牺牲掉一部分缓存。
