类加载器

从虚拟机的角度来看只有两种类型的加载器:

  • 一种是启动类加载器（boostrap classloader）,这个加载器是由 C++实现的，作为虚拟机的一部分。
  • 另一个就是其他的加载器，这些加载器由 java 语言实现，独立于 java 虚拟机，并且全都继承自 java.lang.ClassLoader.

从用户的角度来看, 存在多种不同功能的加载器:

  • 启动类加载器

引导类加载器负责加载基本 Java API，包括例如 rt.jar。 它只加载在引导类路径中找到的具有更高信任级别的类; 因此，它会跳过为大部分普通类完成验证。

  • 扩展类加载器

它负责加载<JAVA_HOME>/lib/ext 目录下或者由系统变量-Djava.ext.dir 指定位路径中的类库。

  • 系统类加载器

它负责加载系统类路径 java -classpath 或-D java.class.path 指定路径下的类库

  • 用户自定义的加载器

用于一些特殊的原因，例如运行时类的重新加载或者如 tomcat 之类的 web 服务器在不同的类组之间的隔离

双亲委派模式​

下图这种类加载器之间的层次关系，称为类加载器的双亲委派模式。


如果父类加载器可以完成类加载任务，就成功返回，倘若父类加载器无法完成此加载任务，子加载器才会尝试自己去加载，这就是双亲委派模式。

它不是一种强制性的模式，而是 java 设计者推荐给开发者的一种类加载实现方式。

优点​

采用双亲委派模式的是好处是 Java 类随着它的类加载器一起具备了一种带有优先级的层次关系，通过这种层级关可以避免类的重复加载，当父亲已经加载了该类时，就没有必要子 ClassLoader 再加载一次。其次防止恶意覆盖 Java 核心 API。

破坏双亲委派模式​

  • 在双亲委派模式出来之前，用户继承 ClassLoader 就是为了重写 loadClass 方法，但双亲委派模式需要这个方法，所以 1.2 之后添加了 findClass 供以后的用户重写
  • 如果基础类要调回用户的代码，如 JNDI/JDBC 需要调用 ClassPath 下的自己的代码来进行资源管理，Java 团队添加了一个线程上下文加载器，如果该加载器没有被设置过，那么就默认是应用程序类加载器
  • 为了实现代码热替换，OSGi 是为了实现自己的类加载逻辑，用平级查找的逻辑替换掉了向下传递的逻辑。但其实可以不破坏双亲委派逻辑而是自定义类加载器来达到代码热替换。

更快的类加载​

从版本 5.0 开始，HotSpot JVM 中引入了一个名为类数据共享（CDS）的功能。

在 JVM 的安装过程中，安装程序会将一组关键 JVM 类（如 rt.jar）加载到内存映射的共享存档中（memory-mapped shared archive）。

CDS 减少了加载这些类所需的时间，从而提高了 JVM 的启动速度，并允许在 JVM 的不同实例之间共享这些类，从而减少内存占用。
