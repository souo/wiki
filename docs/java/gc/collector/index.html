垃圾收集器

serial 收集器​

  • 这个收集器是一个单线程的收集器
  • MinorGC 和 Full GC 都会 Stop The world
  • 新生代使用复制算法
  • 老年代使用 标记-压缩算法
  • 在对停顿时间要求不高的客户端应用
  • 同一台机器运行大量 JVM 实例时，也常使用。垃圾收集时只占用一个 CPU，对其他 JVM 干扰较小。

parallel 收集器：吞吐量优先​

  • 与 serial 收集器，收集方式一致，然而 minor GC 和 Full GC 都是并行的，会用到所有的处理器资源。
  • 注意，这个收集器在早期版本在老年代是串行的，引入 parallel old 收集器后才改变
  • 与 serial 相比改善了垃圾收集的整体效率，从而改善了应用的吞吐量

CMS 收集器： 低延迟为先​

  • 对于很多应用，快速响应比端到端的吞吐量更重要。
  • 为了应对这种情况，HopSpot vm 引入了 CMS（并发标记清除）收集器。
  • 管理新生代的方式与 serial 和 parallel 收集器一致
  • 在老年代尽可能的并发，每个垃圾收集周期只有 2 次短的停顿

CMS 垃圾收集器是基于标记清除算法实现的，目前主要用于老年代垃圾回收。CMS 收集器的 GC 周期主要由 7 个阶段组成，其中有两个阶段会发生 stop-the-world，其它阶段都是并发执行的。

  • （STW） 开始有一个短的停顿，称为初始标记，它标记了从外部（GC Roots）直接可达的老年代对象。
  • 然后在并发标记阶段，标记所有从这些对象可达的存活对象。
  • （STW）因为在标记阶段，应用可能正在运行并更新引用，并发阶段结束时，未必所有存活对象都能被标记。为了应对这种情况需再次停顿，称为重新标记（Remark）
  • 为了进一步减少重新标记的工作量，CMS 收集器引入了并发预清除（pre-cleaning）阶段。预清除在标记完成之后和在重新标记之前，完成一些原本要在 Remark 阶段完成的工作。
  • 最后阶段时并发清除，清除整个 java 堆，释放没有迁移的垃圾对象。

CMS的优缺点：

  • CMS 与前两个垃圾收集器相比，需要更大的 java 堆。
  • CMS 的周期时间长于 Stop-The-world 垃圾收集器所有的时间。
  • 只有在清除阶段，垃圾才会真正的被回收。标记阶段仍然允许分配对象，在标记阶段老年代的占用可能会增加，只有在清除阶段才会减少
  • 尽管垃圾收集器确保在标记阶段可以标记所有存活对象，但实际上垃圾收集器无法保证找出所有的垃圾对象。标记阶段称为垃圾的对象，在周期内可能被收集也可能不被收集。如果没有则在下一次收集。
  • 缺乏压缩，形成空间碎片
  • 在回收周期内，如果尚未会收到足够的空间，老年代满了，CMS 就会退而求其次采用代价昂贵的 Stop-The-World 进行空间压缩，就像 parallel 和 serial 那样
  • 与 paralle 相比，CMS 的老年代停顿变短了，但代价时新生代停顿略微拉长、吞吐量有所降低，heap 的大小有所增长。

这里有更详细的解释

G1​

Garbage-First（G1）收集器是一种服务器式垃圾收集器，适用于具有大容量存储器的多处理器机器。它以高概率满足垃圾收集（GC）暂停时间目标，同时实现高吞吐量。

  • 和 CMS 一样可以与应用程序并发执行
  • 空闲空间压缩，没有长时间 GC 引起的暂停时间
  • 不牺牲很多吞吐量性能。
  • 不需要更大的 Java 堆。

旧的垃圾收集器（serial, parallel, CMS）都将堆构建为三个固定内存大小的部分：年轻代，老年代和永久代 ￼

G1 收集器采用不同的方法。￼

  • 堆被分区为一组大小相等的堆区域，每个区域都是一个连续的虚拟内存区域。 某些区域集被赋予与旧收集器中相同的角色（eden，survivor，old），但它们没有固定的大小。 这为内存使用提供了更大的灵活性。
  • 执行垃圾收集时，G1 以类似于 CMS 收集器的方式运行。
  • G1 执行并发全局标记阶段以确定整个堆中对象的活跃度。
  • 在标记阶段完成之后，G1 知道哪些区域基本上是空的。它首先收集这些区域，这通常会产生大量的空闲空间。这就是为什么这种垃圾收集方法称为 Garbage-First。
  • G1 使用暂停预测模型来满足用户定义的暂停时间目标，并根据指定的暂停时间目标选择要收集的区域数。
  • 被 G1 标记为适合回收的 heap 区将使用疏散(evacuation)的方式进行垃圾回收。
  • G1 将一个或多个 heap 区域中的对象拷贝到其他的单个区域中,并在此过程中压缩和释放内存.
  • 这种疏散在多处理器上并行执行，以减少暂停时间并提高吞吐量
  • 因此,每次垃圾收集时, G1 都会持续不断地减少碎片, 并且在用户给定的暂停时间内执行.
  • 注意：G1 具有并发（与应用程序线程一起运行，例如，细化，标记，清理）和并行（多线程，例如，stop the world)）阶段。 Full GC 仍然是单线程的，但如果调整得当，您的应用程序应该避免使用 Full GC。

G1 分为 Young GC、Mix GC 以及 Full GC

G1 Young GC 主要是在 Eden 区进行，当 Eden 区空间不足时，则会触发一次 Young GC。将 Eden 区数据移到 Survivor 空间时，如果 Survivor 空间不足，则会直接晋升到老年代。此时 Survivor 的数据也会晋升到老年代。Young GC 的执行是并行的，期间会发生 STW。

当堆空间的占用率达到一定阈值后会触发 G1 Mix GC（阈值由命令参数 -XX:InitiatingHeapOccupancyPercent 设定，默认值 45），Mix GC 主要包括了四个阶段，其中只有并发标记阶段不会发生 STW，其它阶段均会发生 STW。

G1 和 CMS 主要的区别在于：

  • CMS 主要集中在老年代的回收，而 G1 集中在分代回收，包括了年轻代的 Young GC 以及老年代的 Mix GC； = G1 使用了 Region 方式对堆内存进行了划分，且基于标记整理算法实现，整体减少了垃圾碎片的产生；
  • 在初始化标记阶段，搜索可达对象使用到的 Card Table，其实现方式不一样。

在垃圾回收的时候都是从 Root 开始搜索，这会先经过年轻代再到老年代，也有可能老年代引用到年轻代对象，如果发生 Young GC，除了从年轻代扫描根对象之外，还需要再从老年代扫描根对象，确认引用年轻代对象的情况。

这种属于跨代处理，非常消耗性能。为了避免在回收年轻代时跨代扫描整个老年代，CMS 和 G1 都用到了 Card Table 来记录这些引用关系。只是 G1 在 Card Table 的基础上引入了 RSet，每个 Region 初始化时，都会初始化一个 RSet，RSet 记录了其它 Region 中的对象引用本 Region 对象的关系。

收集器历史​

  • Serial 收集器（1.3 引入）时 HotSpotVm 的第一个收集器
  • Java 1.4.2 中引入了 Parallel 收集器（只有新生代是并行的，而老年代是 Serial old 收集器）和 CMS（新生代是并行的，并发阶段分成了若干步骤）
  • Java5 update6 引入 Paralle old 收集器是 Parallel 收集器的老年代版本。
  • Java5 update6 和 java5 updata 7 分别实现了 CMS 并发标记和并发清除的并行。
  • java 6 update 20 引入 G1 收集器
